1. Анализ (Reconnaissance)
При анализе приложения были выявлены следующие ключевые точки:

JWT для сессий: Используется алгоритм RS256. Публичный ключ доступен по адресу /.well-known/jwks.json.

Скрытый эндпоинт: Найдена страница /admin, доступная только пользователям с ролью admin.

Функция Restore: В админ-панели реализована функция восстановления бекапа, принимающая данные в формате Base64.

2. Уязвимость №1: JWT JWK Injection
В ходе изучения механизма проверки JWT (или через исходный код app.py) обнаружена поддержка параметра jwk в заголовке токена. Сервер доверял публичному ключу, переданному внутри самого JWT, вместо того чтобы использовать только доверенный локальный ключ.

Эксплуатация:

Сгенерирована собственная пара RSA-ключей.

Создан поддельный токен с sub: test и role: admin.

В заголовок (header) токена вставлен наш публичный ключ в формате JWK.

Токен подписан нашим приватным ключом.

Результат: Получен доступ к панели администратора /admin.

3. Уязвимость №2: Python Pickle Deserialization (RCE)
Форма восстановления бекапа на странице /admin/restore принимает Base64-строку и обрабатывает её с помощью pickle.loads(). Библиотека pickle небезопасна, так как позволяет выполнять произвольный код при десериализации через магический метод __reduce__.

Эксплуатация:

Написан Python-скрипт, создающий вредоносный объект.

В методе __reduce__ использован subprocess.check_output(['cat', '/flag.txt']).

Объект сериализован и закодирован в Base64.

Payload отправлен в форму "Backup blob".

Результат: Сервер выполнил команду и вывел содержимое файла флага в ответе.

Flag: CUTS{jwk_h34d3r_1nj3ct10n_plu5_p1ckl3_rc3}
